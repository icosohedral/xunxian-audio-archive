# 游戏音频资料库网站设计方案

> 文档状态：设计方案与本地原型实施记录（v1.2）  
> 资源盘点日期：2026-07-19  
> 项目目录：`xunxian_resource`  
> 本文基于 `original_data/` 的实际文件进行修订，不再沿用粗略数量估算。

## 一、结论与核心决策

本项目适合采用“静态前端 + 私有对象存储 + 轻量音频网关”的方案：

```text
                        ┌─> Cloudflare Pages
浏览器 ── www.example.com ┤   HTML / CSS / JS / catalog JSON
                        │
                        └─> media.example.com（Cloudflare Worker）
                              │
                              └─> Cloudflare R2 私有 Bucket
                                  仅保存网页试听衍生文件
```

其中：

- Cloudflare Pages 负责网站和公开的检索目录，不直接存放数千个音频文件。
- Worker 负责签名校验、路径白名单、Range 请求、缓存头和 R2 读取。
- 本地原型使用同一套 Hono 路由和 `FsAudioStore` 直接读取 `original_data/`，生产环境只替换为 `R2AudioStore`。
- R2 保持私有，关闭公开的 `r2.dev` 访问，只通过 Worker 的 R2 Binding 读取。
- `original_data/` 是不可变原始归档，不参与网页构建，也不直接上传为公开试听文件。
- 第一版不引入数据库、账户系统和后台管理；分类信息通过可版本控制的规则和覆盖表维护。
- BGM 使用单一全局播放器；音效使用独立试听控制，避免音效点击打断 BGM。
- 公开目录与内部构建清单分离，公开目录不包含源文件校验值和本地路径。

这套方案能够覆盖当前 6,414 个音频文件。项目的主要难点并非存储容量，而是文件分类、稳定标识、重复项处理、移动端播放行为和访问控制边界。

## 二、实际资源盘点

### 2.1 音频资源

| 目录 | 数量 | 格式 | 文件体积 | 总时长 | 主要特征 |
| --- | ---: | --- | ---: | ---: | --- |
| `original_data/music/` | 148 | 148 OGG Vorbis | 218.19 MiB | 4.460 小时 | 全部双声道；143 个为 44.1 kHz |
| `original_data/sound/` | 6,266 | 4,706 OGG Vorbis；1,560 WAV PCM | 368.31 MiB | 4.864 小时 | 6,204 个单声道；中位时长 1.934 秒 |
| 合计 | 6,414 | 4,854 OGG；1,560 WAV | 586.50 MiB | 9.324 小时 | 当前最大单文件约 4.92 MiB |

说明：

- `du` 显示 `music/` 与 `sound/` 的磁盘占用约 599 MiB；表中使用文件字节数换算，二者口径不同。
- 整个 `original_data/` 约 737 MiB，还包含 GUI 图片、技能文本和一个约 100 MiB 的可执行文件；这些不属于音频网站 MVP。
- BGM 平均时长为 108.49 秒，中位时长为 111.83 秒，只有少量曲目达到 3～5 分钟。因此原稿“每首约 3～4 分钟”的估算应删除。
- 音效平均时长为 2.795 秒，90% 不超过 5.8 秒，99% 不超过 13.1 秒。

### 2.2 其他资源

| 目录 | 内容 | 数量/体积 | 当前处理方式 |
| --- | --- | --- | --- |
| `original_data/gui/` | PNG 图标 | 538 个，约 6.3 MiB | 暂不纳入 MVP，可在角色/物品页阶段使用 |
| `original_data/mb/skill/` | 技能配置文本 | 100 个，约 32 MiB | 暂不直接导入；后续可用于技能名和标签映射 |
| `original_data/xunxian_resource.exe` | 原工具或程序 | 约 100 MiB | 不进入网站构建、部署和公开仓库 |

### 2.3 已发现的数据质量问题

1. `music/` 和 `sound/` 之间存在 5 组同名 OGG；其中至少 2 组是字节完全相同的重复文件。
2. `sound/` 中有多条 1～2 分钟的文件，名称和时长都更像 BGM，例如 `xx_field_gxt01.ogg`、`world_bf_bingchuan01.ogg`、`xx_csg_day.ogg`。它们需要进入人工复核队列，不能只按所在目录决定类型。
3. `sound/` 中存在 `juese_zhujue_houzi_daxiao.ogg` 与同名 WAV。两者可能是转码版本，也可能内容有差异，不能用“不含扩展名的 basename”作为唯一 ID。
4. 当前所有音频文件名均为 ASCII 且不含空格，适合自动化处理；但名称大量使用缩写和拼音，直接展示的可读性较弱。
5. 高频前缀包括 `npcyy`、`npcjn`、`guaiwu`、`juese`、`npcskl`、`boss`、`zdjn`、`pet`、`jiemian`、`huanjing` 等，可以作为初始自动分类依据，但必须允许人工覆盖。

因此，导入流程必须输出：重复报告、同名冲突报告、超长音效报告、无法解析报告和待分类报告。

## 三、项目目标与非目标

### 3.1 项目目标

- 在桌面端和移动端快速搜索、筛选和试听 BGM/音效。
- 让 6,000 余条数据仍能保持快速首屏、平滑滚动和即时响应。
- 从原始目录一键生成试听文件、公开目录和质量报告。
- 原始资源、生成物和人工元数据相互分离，做到可重复构建。
- 在不运行传统服务器和数据库的前提下完成低成本部署。
- 对明显盗链、批量扫描和异常请求进行基础限制，并保留进一步加强访问控制的路径。

### 3.2 非目标

MVP 不提供：

- 原始无损文件下载；
- 音频 DRM 或绝对防提取；
- 用户注册、跨设备收藏和评论；
- 在线编辑元数据的后台；
- 自动识别所有角色、地图和技能中文名；
- 对商业游戏资源的版权授权判断。

浏览器要播放音频就必须获得可解码的数据，因此签名 URL 只能降低长期盗链和批量滥用，不能阻止有能力的访问者保存已播放的试听文件。若需要真正的成员访问控制，应增加 Cloudflare Access 或自有登录鉴权。

│   ├── manifests/
│   │   └── private-manifest.json

├── tests/
│   └── fixtures/audio/            # 少量测试音频，不依赖完整资料库
├── original_data/                 # 不可变原始归档，不由脚本改写
│   ├── music/
│   ├── sound/

│   └── mb/

约束：

- `original_data/` 只读；禁止转码脚本覆盖源文件。
- `generated/` 加入 `.gitignore`，可随时删除后重建。
- `apps/web/public/data/` 中只放前端需要的公开字段。
- 私有构建清单包含源路径、哈希和 R2 Key，但不随 Pages 部署。
- 音频不放入 `apps/web/public/audio/`。本地由 `FsAudioStore` 读取 `original_data/`，生产环境统一进入 R2。
- 前端只能依赖 `packages/contracts` 中的公开契约，不直接读取本地文件系统或 R2。
- Node.js 与 Worker 使用不同入口，防止 Node 专属模块被意外打入 Cloudflare Bundle。

## 五、技术选型

### 5.1 前端

推荐：Vite + React + TypeScript。

选择理由：

- 页面状态、筛选条件、全局播放器和虚拟列表已经超过纯静态 HTML 的舒适维护范围。
- React 的资料、示例、测试工具和 AI 训练覆盖更广，适合以 vibecoding 为主要开发方式。
- TypeScript 可约束目录 Schema，减少脚本与前端字段不一致的问题。
- 数据量只有约 6,400 条，无需服务端搜索；浏览器内存检索足够。

状态管理保持简单：全局 BGM 播放器使用 `PlayerProvider + useReducer`，搜索和分页状态写入 URL Query，音量、播放模式和收藏写入 `localStorage`。MVP 不引入 Redux。

### 5.2 构建与自动化

- pnpm workspace：在一个仓库中管理前端、音频服务和共享契约，不在 MVP 阶段引入 Turborepo。
- TypeScript：前端、服务端、扫描、分类、目录生成、报告和上传编排统一使用一种语言。
- Zod：作为目录 JSON、API 返回值和配置文件的唯一运行时 Schema；生成端和消费端都必须验证。
- Hono：以标准 `Request`/`Response` API 实现音频路由，通过 Node.js Adapter 在本地运行，通过 Worker 入口部署到 Cloudflare。
- FFmpeg/FFprobe：转码和标准音频元数据读取。
- Wrangler：Worker 部署、R2 Binding、Secret 和 R2 上传。
- Vitest：单元和集成测试；后续用 Playwright 覆盖真实浏览器播放流程。
- GitHub Actions 或本地脚本：在后续阶段实现可重复发布。

本机当前环境未发现 `ffprobe`，正式实现自动化前应把 FFmpeg 作为明确的开发依赖，并在脚本启动时检查版本；缺失时应立即失败并给出安装说明。

### 5.3 Cloudflare

- Pages：静态网站和公开 JSON。
- R2 Standard：试听音频对象；当前规模远低于 10 GB 免费存储额度。
- Worker：签名与私有 R2 读取。
- WAF Rate Limiting：针对签名接口和音频路径的滥用控制。
- 可选 Bot Fight Mode：仅在观察到机器人流量后开启并验证是否误伤音频请求。

### 5.4 面向 AI Vibecoding 的工程原则

本项目优先优化“AI 每次能快速理解并安全修改”，而不是追求最少文件或最少代码：

- 单语言、单仓库、固定目录和少量统一命令，避免 AI 在多套工具链之间切换。
- 共享 Schema 优先于手写重复类型；字段改变时应由 TypeScript、Zod 和测试立即暴露遗漏。
- 业务路由与存储实现分离。本地使用 `FsAudioStore`，生产使用 `R2AudioStore`，前端接口保持不变。
- 原始资料、人工元数据和生成物分离；任何生成结果都必须可从命令重新构建。
- 测试只使用少量 `tests/fixtures/audio/`，日常修改不扫描或哈希完整的 586.50 MiB 音频库。
- 一个模块只承担一种职责，避免创建“扫描、转码、上传、生成 JSON 全部混在一起”的巨型脚本。
- 新依赖必须解决明确问题；MVP 不引入数据库、GraphQL、微服务、Docker 或复杂状态管理。

根目录 `AGENTS.md` 至少写明以下不可违反的规则：

```text
- 禁止修改 original_data
- 禁止把原始或生成音频提交到 Git
- 所有公开目录必须通过 Zod 校验
- ID 必须根据完整相对路径（含扩展名）生成
- 所有音频路径必须规范化并经过白名单检查
- BGM 与音效使用不同播放器
- sound 页面禁止一次渲染全部条目
- 完成功能前必须运行 pnpm check
```

## 六、音频处理策略

### 6.1 原始文件

原始 OGG 和 WAV 永久保留在 `original_data/`，用于：

- 重新转码；
- 比对重复文件；
- 修正元数据；
- 未来制作更高质量或不同编码版本。

原始文件不进入 Pages，不以原始文件名直接暴露下载入口。

### 6.2 MVP 推荐输出

首选试听格式为 Opus，兼容回退为 MP3：

| 类型 | Opus | MP3 回退 | 声道策略 |
| --- | --- | --- | --- |
| BGM | 96～112 kbps | 128 kbps | 保持立体声 |
| 普通音效/语音 | 48～64 kbps | 80～96 kbps | 源为单声道则保持单声道 |
| 环境长音效 | 64～80 kbps | 96 kbps | 按源文件保留声道 |

执行建议：

1. 第一轮先生成 Opus，并对目标浏览器做兼容性测试。
2. 仅当目标设备无法播放 Opus 时加载 MP3 回退，不要双份预加载。
3. 所有输出统一响度并非 MVP 必需；过度标准化会改变原游戏资源的相对音量。第一版只检测削波和异常静音，后续再决定是否做响度归一化。
4. 对极短音效保留 20～50 ms 的头尾静音，避免转码裁剪造成爆音；是否裁剪必须由听感测试决定。
5. 输出时清除不需要的元数据，并显式设置 MIME：Opus 为 `audio/ogg; codecs=opus`，MP3 为 `audio/mpeg`。

按现有总时长粗估：

- BGM 以 96 kbps Opus 输出约 193 MB（十进制）；
- 音效以 64 kbps Opus 输出约 140 MB；
- 全部 Opus 约 333 MB；
- 若同时保留上述 MP3 回退，两套试听文件合计约 0.8 GB，仍远小于 R2 Standard 的免费存储额度。

最终大小应以转码报告为准，因为 VBR、静音和编码器设置会造成偏差。

### 6.3 输出文件名与对象 Key

不要继续以 basename 作为唯一标识。建议：

```text
audio/v1/music/8f/8f32c0e4d2ab.opus
audio/v1/sound/14/14a9707fb88c.opus
```

- ID 由规范化后的“源相对路径 + 扩展名”生成稳定哈希，建议取 SHA-256 前 12～16 个十六进制字符。
- R2 Key 中加入发布版本，便于不可变缓存和回滚。
- 同名 WAV/OGG 会得到不同 ID；确认内容等价后，再通过去重表把它们指向同一试听对象。
- 发布后不覆盖同一个 Key 的内容；内容变化时生成新版本或新内容哈希。

### 6.4 转码幂等性

私有 Manifest 为每个源文件记录：

- 源相对路径；
- 源文件 SHA-256；
- 源大小与修改时间；
- 转码参数版本；
- 输出文件 SHA-256；
- 输出大小和 R2 Key。

只有源哈希或转码参数变化时才重新转码，避免每次全量处理 6,414 个文件。

## 七、元数据与目录设计

### 7.1 目录拆分

不建议只生成一个不断变大的 `tracks.json`。MVP 使用三份文件：

- `catalog-summary.v1.json`：版本、计数、生成时间和分类统计；
- `music.v1.json`：BGM 页面按需加载；
- `sound.v1.json`：音效页面按需加载。

浏览器不扫描 R2，也不调用 R2 List API。所有可见条目均来自构建时目录。

### 7.2 公开条目 Schema

```json
{
  "id": "snd_14a9707fb88c",
  "kind": "sound",
  "title": "npcyy_huangsanye_siwang",
  "originalName": "npcyy_huangsanye_siwang.ogg",
  "durationMs": 2184,
  "category": "voice",
  "subcategory": "npc",
  "tags": ["npc", "death"],
  "character": "huangsanye",
  "locale": "zh-CN",
  "reviewStatus": "auto",
  "audio": {
    "preferredKey": "audio/v1/sound/14/14a9707fb88c.opus",
    "variants": [
      {
        "key": "audio/v1/sound/14/14a9707fb88c.opus",
        "format": "opus",
        "mimeType": "audio/ogg; codecs=opus"
      },
      {
        "key": "audio/v1/sound/14/14a9707fb88c.mp3",
        "format": "mp3",
        "mimeType": "audio/mpeg"
      }
    ]
  },
  "technical": {
    "sampleRate": 44100,
    "channels": 1
  }
}
```

约定：

- 时长统一使用整数毫秒，避免浮点比较问题。
- `title` 是展示标题；没有人工标题时使用去扩展名后的原文件名。
- `originalName` 用于精确检索和排查，但不等于音频对象 Key。
- `reviewStatus` 取 `auto`、`reviewed`、`needs_review`。
- `audio.preferredKey` 是默认试听对象；前端可通过 `variants` 检查兼容回退。
- 本地目录可把现有 OGG/WAV 写成 `source/music/...` 或 `source/sound/...` Variant，生产目录则使用 Opus/MP3 的版本化 Key；两者遵守同一 Schema。
- `audio` 中的 Key 是对象标识，不是可直接访问 URL；前端播放前统一请求 `/api/audio-url`。
- 空字段省略，不用空字符串占位。

### 7.3 私有 Manifest

私有 Manifest 额外包含：

- 本地源路径；
- 源/输出 SHA-256；
- 源编码、比特率、声道、采样率和文件大小；
- 转码命令版本；
- 上传状态、R2 ETag 和最后上传时间；
- 重复组 ID 和分类规则命中信息。

此文件只用于构建、审计和增量上传，不随网站发布。

## 八、分类、标签与人工修订

### 8.1 初始一级分类

建议将 `sound` 划分为：

- `voice`：NPC、角色、怪物、Boss、宠物语音；
- `skill`：玩家、NPC、Boss、宠物技能；
- `battle`：攻击、受击、死亡、武器、战斗反馈；
- `ui`：界面、按钮、商城、进度条、拾取/丢弃；
- `environment`：环境、天气、水、村庄、场景氛围；
- `movement`：脚步、坐骑、移动；
- `system`：系统提示、功能反馈；
- `music_candidate`：位于 `sound/` 但疑似音乐的长音频；
- `unknown`：规则无法可靠判断的文件。

### 8.2 前缀规则示例

```yaml
rules:
  - prefixes: [npcyy, vo, vonpc, yy]
    category: voice
    subcategory: npc
  - prefixes: [guaiwu, boss, npcboss]
    category: voice
    subcategory: monster
  - prefixes: [npcjn, npcskl, boskl, bossjn, petskl, zdjn, jineng]
    category: skill
  - prefixes: [jiemian, jindutiao, shiqu, diuqi]
    category: ui
  - prefixes: [huanjing, tianqi, shui, cunzhuang]
    category: environment
  - prefixes: [step, qichong]
    category: movement
```

规则必须记录命中的规则 ID。人工 `overrides.json` 的优先级高于自动规则，重新扫描时不能覆盖人工结果。

### 8.3 自动复核条件

满足以下任一条件时标记为 `needs_review`：

- `sound` 中时长超过 30 秒；
- 跨目录 basename 冲突；
- 同 basename 存在多种格式；
- 同内容哈希对应多个路径；
- 无法解析格式或时长；
- 分类为 `unknown`；
- 采样率、声道或时长明显异常；
- 文件近似静音或检测到削波。

## 九、前端信息架构与功能

### 9.1 页面结构

```text
/
├── /music        BGM 列表与持续播放器
├── /sounds       音效检索与快速试听
├── /about        数据来源、使用说明、版权与版本
└── /404
```

首页展示实际统计数据、BGM/音效入口、最近更新和基本使用说明，不把 6,000 条数据全部加载到首屏。

### 9.2 通用检索

支持：

- 原文件名、展示名、别名、角色、分类和标签检索；
- 大小写不敏感；
- `_`、`-` 和连续空白等价；
- 精确文件名匹配优先；
- 搜索词和筛选条件同步到 URL Query，刷新或分享链接后可恢复；
- 输入防抖建议 100～200 ms；6,400 条数据无需远程搜索服务。

### 9.3 BGM 页面

功能：

- 搜索、分类和标签筛选；
- 播放/暂停、上一首、下一首；
- 顺序、随机、单曲循环；
- 进度拖动、当前时间/总时长、音量和静音；
- 当前队列和当前播放项高亮；
- 媒体会话信息（浏览器支持时显示系统级标题和控制）；
- 播放状态跨站内路由保持，但默认不跨刷新自动播放。

实现上使用一个全局 `HTMLAudioElement`。切换页面不得销毁它。遵守浏览器自动播放限制：首次播放必须由用户手势触发。

### 9.4 音效页面

功能：

- 点击行或播放按钮立即试听；
- 再次点击当前项时从头重播；
- 默认新音效会停止上一条，避免大量重叠；
- 可选“允许重叠”模式使用最多 4～8 个播放器的对象池，并限制并发数量；
- 快速筛选一级分类、子分类、角色和标签；
- 显示原文件名、时长、分类和人工复核状态。

BGM 和音效使用不同的播放器实例。试听音效不应暂停 BGM，但可提供“试听时自动压低 BGM 音量”的可选设置。

### 9.5 大列表性能

推荐 MVP 使用“内存筛选 + 每页 100 条分页”，比立即引入虚拟列表更简单、可访问性更好。若实际交互需要连续滚动，再替换为虚拟列表。

无论采用何种方式：

- DOM 中不同时存在 6,266 行；
- 不为每一行创建永久 Audio 实例；
- 不预加载所有音频，列表项统一使用 `preload="none"`；
- 只对当前项及用户即将播放的项目请求签名 URL；
- 目录 JSON 使用 Brotli/Gzip 和长期缓存。

### 9.6 收藏与最近播放

可作为低成本的 1.1 版本功能：

- 使用 `localStorage` 保存 ID 列表；
- 保存目录版本，条目消失时自动清理；
- 明确提示仅保存在当前浏览器，不承诺跨设备同步。

## 十、统一音频服务与 R2 接口

本地和生产环境共用 `apps/media/src/app.ts` 中的 Hono 路由，通过依赖注入切换存储实现：

```text
React 前端
    │
    └─> Hono 路由（固定 API 契约）
            │
            ├─ 本地：FsAudioStore -> original_data/
            └─ 生产：R2AudioStore -> 私有 R2 Bucket
```

建议定义最小存储接口：

```ts
interface AudioStore {
  getMetadata(key: string): Promise<AudioMetadata | null>
  get(key: string, range?: ByteRange): Promise<AudioObject | null>
}
```

`app.ts` 只负责路由、参数校验、路径规范化、Range 和错误响应，不直接导入 Node 文件系统或 R2 类型。`node.ts` 注入 `FsAudioStore`，`worker.ts` 注入 `R2AudioStore`。这样本地原型迁移到 Cloudflare 时只替换存储和签名策略，不重写前端与核心路由。

### 10.1 建议接口

```text
GET  /api/audio-url?key=<url-encoded-audio-key>
HEAD /audio/<object-key>?e=<unix-seconds>&sig=<base64url-hmac>
GET  /audio/<object-key>?e=<unix-seconds>&sig=<base64url-hmac>
```

`/api/audio-url` 返回：

```json
{
  "url": "https://media.example.com/audio/v1/sound/14/14a9707fb88c.opus?e=1784449200&sig=...",
  "expiresAt": 1784449200
}
```

建议有效期 10～20 分钟。前端在到期前约 60 秒重新获取；同一条目在有效期内复用 URL，避免每次播放都产生签名请求。

本地模式保持相同接口，但返回无需签名的 `/audio/...` URL，并将 `expiresAt` 设为 `null`；生产模式返回短期签名 URL。前端只依赖共享响应 Schema，不判断底层是文件系统还是 R2。

### 10.2 签名内容

HMAC-SHA-256 至少覆盖：

```text
NORMALIZED_METHOD + "\n" + HOST + "\n" + NORMALIZED_PATH + "\n" + EXPIRES
```

要求：

- Secret 通过 Worker Secret 配置，不写入仓库或前端。
- 使用 `crypto.subtle.verify()`，不要自己进行普通字符串比较。
- 签名使用 Base64URL，避免 `+`、`/`、`=` 带来的 URL 编码歧义。
- 为了让同一个播放 URL 可响应浏览器的 `GET` 和 `HEAD`，签名时可将两者的 `NORMALIZED_METHOD` 都规范化为 `GET`；其他方法一律拒绝。
- 拒绝过期时间过远、非数字、重复 Query 参数和签名长度异常的请求。
- 先验证签名，再读取缓存或 R2。

若签名接口对任何访客开放，它只是防盗链措施，不是身份认证。若需私有站点，签名接口本身必须置于 Cloudflare Access 或应用登录之后。

### 10.3 路径与方法校验

Worker 只接受：

- `GET` 和 `HEAD`；
- 规范化后以 `audio/v1/music/` 或 `audio/v1/sound/` 开头的 Key；
- 白名单后缀 `.opus`、`.ogg`、`.mp3`；
- 不含 `..`、反斜杠、空字节、重复斜杠或解码后二次变化的路径；
- 长度在合理上限内的路径。

不提供 Bucket List，不根据错误返回内部路径。不存在或无权限的对象统一返回简短的 `404`/`403`。

### 10.4 Range、条件请求与响应头

音频拖动依赖字节范围请求。Worker 应正确处理：

- `Range` 与 `If-Range`；
- `If-None-Match`；
- 单范围请求；多范围请求可明确返回 `416`，MVP 无需生成 multipart range；
- `HEAD` 不返回 Body，但返回与 `GET` 一致的元数据头。

响应至少设置：

```text
Content-Type: audio/ogg 或 audio/mpeg
Content-Length: ...
Content-Range: bytes start-end/total        # 仅 206
Accept-Ranges: bytes
ETag: ...
Cache-Control: public, max-age=<签名剩余秒数>, s-maxage=86400, immutable
Content-Disposition: inline
X-Content-Type-Options: nosniff
```

跨子域部署时，CORS 只允许网站域名，并放行 `Range`，暴露 `Accept-Ranges`、`Content-Length`、`Content-Range` 和 `ETag`。开发和预览域名用环境变量配置，不使用 `*` 与凭据组合。

### 10.5 缓存设计

签名参数会造成缓存碎片，缓存 Key 应使用移除 `e` 和 `sig` 后的规范对象路径，但必须遵守以下顺序：

1. 验证路径、过期时间和 HMAC；
2. 构造不含签名参数的内部缓存 Key；
3. 查询缓存；
4. 未命中时读取 R2；
5. 缓存完整的 `200` 响应；
6. 对 Range 请求从完整缓存对象返回 `206`。

不能把 `206 Partial Content` 直接写入 Workers Cache API。当前最大源文件约 4.92 MiB，MVP 可在缓存未命中时读取并缓存完整衍生文件；实现时仍要设置对象大小上限，防止未来意外上传大文件导致内存压力。

不要在签名验证之前用忽略签名的共享缓存直接回源，否则缓存命中可能绕过授权。上线前必须用冷缓存和热缓存分别测试 `200`、`206`、`304`、`403`、`404` 和 `416`。

## 十一、安全、滥用与版权

### 11.1 R2 与密钥

- R2 Bucket 保持私有并关闭 `r2.dev`。
- Worker 通过 R2 Binding 读取，运行时不需要在代码里保存 S3 Access Key。
- 上传脚本使用权限最小化的 R2 API Token，CI Secret 与 Worker HMAC Secret 分开。
- 生产、预览环境使用不同的 Bucket、域名和 Secret。
- Secret 定期轮换；如需无缝轮换，短期同时接受当前和上一个密钥版本。

### 11.2 限流

当前 Worker 使用 Rate Limiting Binding，以 Cloudflare 提供的客户端 IP 为 Key：签名接口每分钟 30 次，音频接口每分钟 180 次，超限返回带 `Retry-After: 60` 的 `429`。浏览器可能为播放、拖动、重试产生多次 Range 请求，共享 NAT 也会让多个用户表现为同一 IP，因此阈值应结合生产流量继续观察。

推荐分别处理：

- `/api/audio-url`：较严格，优先限制批量签名；
- `/audio/*`：较宽松，主要拦截高速枚举和并发抓取；
- 明显异常时先使用 Managed Challenge，再考虑长时间 Block；
- 排除已验证的友好机器人没有实际必要，因为音频路径不需要搜索引擎抓取。

Rate Limiting Binding 按 Cloudflare 边缘位置独立计数并采用最终一致性，适合缓解滥用，但不是精确计费或全局配额系统。阈值同时记录在 Worker 配置与本节；修改时应一并更新。

### 11.3 Bot 防护

Bot Fight Mode 可作为辅助，但它作用于整个域名且可定制能力有限，可能误伤 API 或自动化检查。建议先观察 Security Events，再在 `media` 独立域名上评估启用；上线后必须测试主流桌面和移动浏览器。

### 11.4 Web 安全头

Pages 建议设置：

```text
Content-Security-Policy: default-src 'self'; media-src 'self' https://media.example.com; connect-src 'self' https://media.example.com; img-src 'self' data:; object-src 'none'; base-uri 'self'; frame-ancestors 'none'
Referrer-Policy: strict-origin-when-cross-origin
X-Content-Type-Options: nosniff
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

具体 CSP 要随前端依赖调整，避免为了省事加入不必要的 `unsafe-inline`。

### 11.5 版权边界

若资源来自商业游戏，技术上的“可部署”不等于拥有公开传播权。上线前应确认授权范围、地域、访问对象和可否提供下载。最低限度：

- 不提供源文件或批量下载入口；
- 只提供压缩试听衍生文件；
- About 页说明资源来源、用途、权利归属和联系/下架渠道；
- 保留按 ID 或分类快速下架的能力；
- 对权利状态不确定的项目优先使用 Access 限制为个人或小范围使用。

## 十二、自动化流水线

### 12.1 建议命令

```text
pnpm dev                         # 同时启动 React 和本地 Hono 服务
pnpm catalog:scan                # 扫描原始资源
pnpm catalog:build               # 分类并生成目录
pnpm audio:transcode             # 生成部署用试听文件
pnpm catalog:validate            # 验证 Schema、引用和冲突
pnpm upload -- --dry-run         # 查看 R2 增量上传计划
pnpm upload
pnpm test
pnpm typecheck
pnpm lint
pnpm check                       # typecheck + lint + test + catalog validate
pnpm build
```

根目录命令是 AI 和人工开发者的唯一推荐入口，避免要求 AI 记忆各子项目命令。每一步应支持单独运行、失败重试；上传和清理必须支持 `--dry-run`。

### 12.2 扫描

扫描范围只允许显式配置的 `original_data/music` 与 `original_data/sound`。读取：

- 相对路径、扩展名和字节数；
- 格式、编码器、时长、声道、采样率和比特率；
- SHA-256；
- 可选的峰值、静音比例和响度检测结果。

任何无法解析的文件都应令校验阶段失败，不可静默跳过。

### 12.3 分类

处理优先级：

```text
人工 overrides > 已审核历史记录 > 自动前缀规则 > 目录默认值
```

分类结果必须可解释，例如记录 `matchedRule: sound-prefix-npcyy`，以便修正规则时定位影响范围。

### 12.4 转码

要求：

- 输出到临时文件，FFmpeg 成功且校验通过后原子替换生成物；
- 限制并发，避免同时转码过多文件耗尽 CPU/文件句柄；
- 捕获完整命令、退出码和 stderr；
- 转码后再次读取时长，允许小范围编码延迟差异；
- 不修改原文件时间戳和内容。

### 12.5 校验与报告

构建失败条件：

- ID 重复；
- R2 Key 重复且内容不同；
- 输出文件缺失、大小为 0 或无法解码；
- 公开目录引用不存在的输出；
- JSON 不符合 Schema；
- 音频时长差异超过设定阈值；
- 公开目录意外包含绝对本地路径或 Secret。

每次生成以下报告：

```text
generated/reports/inventory.json
generated/reports/duplicates.csv
generated/reports/name-collisions.csv
generated/reports/needs-review.csv
generated/reports/transcode-failures.csv
generated/reports/size-summary.md
```

### 12.6 上传与发布

1. 生成并校验新版本音频。
2. 以不可变 Key 增量上传 R2，先 `--dry-run` 查看新增、跳过和删除数量。
3. 抽样通过 Worker 预览域名播放。
4. 最后发布引用新 Key 的目录 JSON 和前端。
5. 保留上一版目录和对象至少一个回滚周期。
6. 通过生命周期或单独清理命令删除不再引用的旧版本；删除必须生成清单并人工确认。

先上传音频、后发布目录，可以避免新页面引用尚不存在的对象。

## 十三、测试与验收标准

### 13.1 数据测试

- 日常测试使用 `tests/fixtures/audio/` 中的少量 OGG/WAV，保证 AI 修改后的反馈速度。
- 扫描结果固定为预期数量，新增/删除必须在报告中显式显示。

- 已知重复项出现在重复报告中。
- `sound` 中超过 30 秒的文件全部进入复核队列。
- 公开 JSON 可通过 Schema 校验，且不包含绝对路径。

### 13.2 播放测试

至少覆盖当前版本的：

- Chrome、Firefox、Safari、Edge；
- iOS Safari 和 Android Chrome；
- 桌面宽屏和窄屏；
- 慢速网络、断网恢复和 URL 过期重签；
- BGM 拖动、上一首/下一首、循环和随机；
- 音效连续点击、重复点击和 BGM 同时播放；
- 后台/锁屏后的媒体会话行为。

### 13.3 统一音频服务测试

- 同一组路径、Range 和错误响应测试必须同时覆盖 `FsAudioStore` 与 `R2AudioStore`。
- 本地 `/api/audio-url` 返回直连 URL 和 `expiresAt: null`，生产模式返回签名 URL 和有效期。
- 合法签名能播放；缺签名、篡改路径、过期签名返回 `403`。
- `../`、编码后的路径穿越、非法后缀和超长路径被拒绝。
- 合法 Range 返回正确 `206`、`Content-Range` 和字节内容。
- 越界 Range 返回 `416`。
- 热缓存不会绕过签名；不同签名可安全命中同一内部对象缓存。
- `HEAD`、`ETag`、`304` 和 CORS 行为正确。
- 不存在对象不泄漏 Bucket 信息。

### 13.4 性能目标

在常见移动网络下建议目标：

- 首页不加载音频目录；
- 进入 BGM/音效页后，目录压缩传输并在可接受时间内完成解析；
- 搜索输入到结果更新通常低于 100 ms；
- 页面 DOM 行数保持在数百以内；
- 点击播放后立即显示加载状态，热缓存音频尽快开始播放；
- Lighthouse 性能、可访问性、最佳实践和 SEO 均纳入 CI 或发布前检查。

性能指标应以实测为准，不在尚未实现时承诺绝对毫秒值。

## 十四、成本与容量判断

以当前规模计算：

- 原始音频约 0.59 GiB，本地归档即可；
- 两种试听格式预计约 0.8 GB；
- 即使保留多个短期版本，也远低于 R2 Standard 当前每月 10 GB 的免费存储额度；
- R2 当前不收取互联网出口流量费，但读取会计入 Class B 操作；
- R2 Standard 当前免费额度为每月 100 万次 Class A、1,000 万次 Class B；
- Workers 免费计划当前为每天 100,000 次动态请求；若站点增长或签名/音频请求频繁，应考虑最低月费的 Paid 计划并设置用量告警。

实际成本主要由请求次数而非 0.8 GB 存储决定。应记录：

- Worker 请求数、错误率和 CPU 时间；
- R2 Class A/Class B 操作；
- 缓存命中率；
- 每日播放次数和 Range 请求数；
- 签名接口与音频接口的 `403`、`404`、`416`、`429` 数量。

不要把免费额度视为永久产品承诺；部署前和每次架构调整时复核官方定价。

## 十五、运维、版本与故障处理

### 15.1 版本

- JSON 根对象包含 `schemaVersion`、`catalogVersion`、`generatedAt`。
- 前端只接受明确支持的 Schema 主版本。
- 音频对象 Key 包含发布版本或内容哈希，缓存可设为不可变。
- About 页显示目录版本，方便用户反馈具体问题。

### 15.2 日志与隐私

- Worker 日志记录请求 ID、状态码、对象 ID、Range 类型和耗时，不记录完整签名 URL。
- IP 等数据只保留排障所需的最短时间，遵守部署地适用的隐私规则。
- 对用户可见的错误信息简短，详细错误只进入受控日志。

### 15.3 回滚

- Pages 保留上一成功部署。
- R2 新版本对象先上传且短期不删除旧对象。
- 回滚只需恢复上一版前端/目录；旧对象仍可被引用。
- HMAC Secret 轮换导致旧 URL 失效是预期行为，不影响重新签名。

### 15.4 下架

维护一个 `disabled-ids.json`：

- 构建时从公开目录移除；
- Worker 可选地立即拒绝对应对象，避免等待页面缓存过期；
- 后续再按清理流程删除 R2 对象。

## 二十、项目目标总结

本项目应被实现为一个轻量、可重复构建、便于持续整理的游戏音频资料库，而不是通用音乐平台。当前真实资源规模为 148 首 BGM 和 6,266 条音效/语音候选文件，总计 6,414 个音频、约 9.32 小时。

方案以不可变原始归档为基础，通过 TypeScript + FFmpeg 自动生成试听衍生文件、公开目录、私有 Manifest 和质量报告。项目采用 pnpm workspace、React/Vite、Hono 和共享 Zod Schema；同一套音频路由在本地注入 `FsAudioStore`，生产环境注入 `R2AudioStore`。前端由 Cloudflare Pages 托管，音频由私有 R2 保存并经 Worker 提供短期签名访问。缓存、Range、路径校验、限流和基础 Bot 防护用于降低成本与滥用风险，但不将签名 URL 错当成 DRM 或身份认证。

第一阶段的重点应放在稳定 ID、数据质量、可重复目录生成、本地正确播放和清晰的 AI 修改边界上；转码与 Cloudflare 部署在本地原型稳定后进行。收藏、波形、中文映射、角色/地图专题和用户系统都可以在这个稳定基础上逐步扩展。
